# Codex 端到端复测记录 2026-04-17

## 目标

复测两条链路下 Codex 使用 firew2oai 的真实表现：

- `Codex -> firew2oai`
- `Codex -> new-api -> firew2oai`

覆盖范围包括最小文本任务、多轮会话、工具调用与 `spawn_agent`。

## 修复摘要

- 修正 Responses API 生命周期事件格式，`response.created` 和 `response.completed` 均使用 `{"type": "...", "response": {...}}` 包装。
- 将 Codex 的 `instructions`、`developer` 上下文、仓库上下文和当前用户任务分区，显式突出 `CURRENT_USER_TASK`。
- 增加文本模型工具桥接：当上游输出 JSON 工具调用时，转换为 Responses `function_call` / `custom_tool_call` item。
- 增加工具输出输入解析，允许下一轮把 `function_call_output` 转回上游可理解的上下文。
- Responses 状态存储改为保留原始 request items 与历史 items，`previous_response_id` 不再仅依赖扁平聊天文本。
- `response.completed` 补充本地估算 `usage`，兼容 new-api 的展示与计费视图。
- 当上游输出未声明工具名或非法参数时，返回显式 `Codex adapter error`，避免静默降级为普通回答。

## 结果矩阵

| 场景 | 链路 | 结果 | 证据 |
| --- | --- | --- | --- |
| 最小文本任务 `只回答 ok` | 直连 | 通过 | JSONL 返回 `agent_message: ok` |
| 最小文本任务 `只回答 ok` | new-api 中转 | 通过 | JSONL 返回 `agent_message: ok`，firew2oai 日志出现 `/v1/responses` |
| `previous_response_id` 多轮恢复 | 包内测试 | 通过 | `TestHandleResponses_PreviousResponseID` 与 `TestHandleResponses_PreviousResponseIDWithToolOutput` 均通过 |
| `response.completed.usage` | 包内测试 | 通过 | stream / non-stream 均断言 `input_tokens`、`output_tokens`、`total_tokens` 非零 |
| 工具任务读取 `README.md` 第一行 | 直连 | 部分通过 | `exec_command` 已实际执行一次 `sed -n '1,8p' README.md`，但后续轮次模型又偏航到未声明工具 |
| 未声明工具防降级 | 直连 | 通过 | deepseek-v3p2 输出 `list_files`、`run_terminal_cmd` 时，firew2oai 返回显式 `Codex adapter error` |
| kimi-k2p5 复杂读取任务 | 直连 | 未通过 | JSONL 仅返回空 `agent_message`，无工具调用，问题位于上游模型行为 |
| 多轮会话暗号回忆 | 直连/new-api | 未通过 | 第一轮 `ok` 成功，恢复会话第二轮转入错误工具调用 |
| `spawn_agent` | 直连 | 未通过 | 触发长回合工具调用，但未稳定收敛到有效 `spawn_agent` 调用与最终答案 |

## 结论

当前版本已经解决 Responses 流闭合、原始 item 历史保存、`previous_response_id` 多轮恢复以及 `usage` 缺失问题。直连与 new-api 中转均能完成最小文本任务，协议级工具回灌也已具备。

真实复杂工具任务与 `subagent` 仍未达到稳定可用标准。当前阻塞点已从 HTTP/SSE 协议转移到上游文本模型的工具遵循度：会出现空输出、编造未声明工具名、或第一轮工具成功后后续轮次再次偏航。firew2oai 已将这类失败显式暴露为 `Codex adapter error`，便于继续针对模型行为做后续治理。
