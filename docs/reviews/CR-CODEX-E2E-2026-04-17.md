# Codex 端到端复测记录 2026-04-17

## 目标

复测两条链路下 Codex 使用 firew2oai 的真实表现：

- `Codex -> firew2oai`
- `Codex -> new-api -> firew2oai`

覆盖范围包括最小文本任务、多轮会话、工具调用与 `spawn_agent`。

## 修复摘要

- 修正 Responses API 生命周期事件格式，`response.created` 和 `response.completed` 均使用 `{"type": "...", "response": {...}}` 包装。
- 将 Codex 的 `instructions`、`developer` 上下文、仓库上下文和当前用户任务分区，显式突出 `CURRENT_USER_TASK`。
- 增加文本模型工具桥接：当上游输出 JSON 工具调用时，转换为 Responses `function_call` / `custom_tool_call` item。
- 增加工具输出输入解析，允许下一轮把 `function_call_output` 转回上游可理解的上下文。
- Responses 状态存储改为保留原始 request items 与历史 items，`previous_response_id` 不再仅依赖扁平聊天文本。
- `response.completed` 补充本地估算 `usage`，兼容 new-api 的展示与计费视图。
- 当上游输出未声明工具名或非法参数时，返回显式 `Codex adapter error`，避免静默降级为普通回答。
- 新增尾部控制块 `AI_ACTIONS_V1` 解析，并保留旧版单 JSON fallback。
- 工具协议运行时约束收紧：结构化/自由格式字段错位显式报错，`tool_choice="required"` 变为硬约束，`parallel_tool_calls=false` 会限制为单调用。
- Responses 工具流补齐 `response.output_item.added -> response.output_item.done` 生命周期，避免工具分支与文本分支状态机分叉。

## 渠道路由复核

针对“new-api 是否走错渠道”的疑问，本轮直接查询生产 `new-api` 数据库：

```sql
select id,name,priority,status
from channels
where id = 106 or name ilike '%firew2oai%';

select id,channel_id,model_name,created_at
from logs
where id >= 159887 and model_name = 'kimi-k2p5'
order by id asc;
```

结果：

- `channel_id=106`
- `name=firew2oai-local`
- `priority=100`
- `status=1`
- 最新复测请求命中 `logs.id=159893`

因此，本轮 `Codex -> new-api -> firew2oai` 失败，不是渠道优先级或路由偏差导致，而是命中 firew2oai 后，上游模型仍输出连续顶层工具 JSON，最终被适配层显式拒绝。

本轮补做了三件事：

1. 将 firew2oai 最新代码重新构建并替换生产容器（`docker compose build firew2oai && docker compose up -d firew2oai`）。

路由核验：

- 说明问题不在“是否走指定渠道”，而在“中转链路下模型行为仍不稳定”。

对照验证：

- 同时抽样直连 `http://127.0.0.1:39527/v1`（不经 new-api）对 `deepseek-v3p2` 复测，`cmd_count=3`，表现与直连基线一致。
- 说明 firew2oai 最新容器本身已生效，new-api 中转链路仍存在额外行为差异。

## 结论

当前版本已经解决 Responses 流闭合、原始 item 历史保存、`previous_response_id` 多轮恢复以及 `usage` 缺失问题。直连与 new-api 中转均能完成最小文本任务，协议级工具回灌也已具备。

但这不等于 Codex 真实可用。结合模型矩阵复测，当前结论应收紧为：

- firew2oai 对 Codex 的协议适配是成立的
- firew2oai 作为 Codex 复杂多轮 Agent 默认模型通道，当前不具备可用性
- `kimi-k2p5` 在直连与 new-api 中转两条链路下都未出现效果提升
- `spawn_agent`、多轮工具链、复杂文件审计等真实任务不应视为可支持场景
